# Prompt 设计建议 — 给 MiniMax-M3 的高亮 prompt

> 目标：基于 `theory-notes.md` 的 10 条规则，给 MiniMax-M3 写出一组可测试的 prompt 模板。
> 范围：单段（paragraph-level）高亮 prompt + 整篇 meta prompt
> 原则：规则可拆解、可 A/B、可解释

---

## 1. 主 prompt 模板（推荐 v1 — 简洁版）

```
你是「眺览」阅读引擎的高亮标注助手。请对下面这段中文文本标注高亮位置。

## 任务
- 输出格式：JSON 数组，每个元素是 {start: number, length: number, reason: string}
- 字符偏移按 UTF-16 code unit 计数（与 JavaScript .slice 一致）
- 不修改原文，只标注

## 高亮规则（按优先级）
1. **专有名词、人名、地名、品牌、术语**：必高亮（认知瓶颈）
2. **低频实词**（非"的/了/是/在/和/也/都"等虚词）：高亮
3. **每段的前 2-3 个字**：必须至少高亮 1 个（段首 topic 锚点）
4. **每段的最后 1-2 个字**：不强求（中文阅读 wrap-up 是反转的，句末是视觉休息点）
5. **相邻高亮**：字符距离 6-10 为佳，不超过 12（眼跳距离效应）
6. **高亮位置**：落在词的左 1/3（ORP 原理），不要高亮在词尾
7. **不要**连续高亮 3+ 个生僻字（中央凹预视会失败）
8. **密度**：高亮总字符数 / 段落长度 = 30-50%

## 输入段落
"""
{paragraph_text}
"""

## 输出（只输出 JSON，不要任何解释）
```

### 关键决策记录

- **不放 any English 例子**：会和 few-shot 模板冲突；few-shot 例子会单独给
- **不写 "highlight" 这个词**：用「标注高亮位置」中文描述，避免和原版 mark 标签混淆
- **不指定 ORP 算法**：只给"靠近词首"的方向，因为 LLM 算精确字符偏左 1/3 不可靠
- **不写"7±2 字"**：LLM 不能精确控制，但能理解"6-10"的范围
- **写明 UTF-16**：MiniMax-M3 偶尔会数错多字节字符，明确约定减少误差

---

## 2. 元 prompt（整篇文档 — 上下文层）

```
你将收到一篇文章的若干段落。任务是逐段标注高亮位置。

## 全局约束
- 每段独立输出 JSON 数组
- 段间不要互相参考标号——每段 start=0 重新计数
- 整篇高亮覆盖率 30-50%（不能某段 80% 另一段 0%）
- 段首段尾处理按各段规则

## 处理顺序
1. 先扫一遍全文，识别所有专有名词/术语/低频词（建一个 candidate set）
2. 逐段标注：先用 candidate set 填必高亮项，再用段首规则补足
3. 用"相邻距离 6-10"规则检查，违规的两两合并或删一个
4. 用"30-50% 密度"检查，过密删低优先级的，过疏补段中实词
```

### 关键决策

- **要求 LLM 先建 candidate set 再标注**：模拟人类"先找关键词再决定高亮"的认知流程
- **明确"两两合并或删一个"**：给 LLM 一个明确的"违规处理"动作，不要让它卡住
- **不要求位置：只要求密度**：精确控制密度 LLM 容易越调越乱

## 3. Few-shot 例子（3 个，覆盖典型场景）

### Example 1: 散文段落（白话文，高密度候选）

**Input**：
```
不必说碧绿的菜畦，光滑的石井栏，高大的皂荚树，紫红的桑椹；也不必说鸣蝉在树叶里长吟，肥胖的黄蜂伏在菜花上，轻捷的叫天子忽然从草间直窜向云霄里去了。
```

**Expected Output**：
```json
[
  {"start": 0, "length": 1, "reason": "段首 topic 锚点"},
  {"start": 3, "length": 2, "reason": "形容词低频实词（碧绿）"},
  {"start": 8, "length": 2, "reason": "名词（菜畦）"},
  {"start": 14, "length": 1, "reason": "名词（石井栏）"},
  {"start": 21, "length": 2, "reason": "名词（皂荚树）"},
  {"start": 27, "length": 2, "reason": "形容词（紫红）"},
  {"start": 32, "length": 2, "reason": "名词（桑椹）"},
  {"start": 40, "length": 2, "reason": "名词（鸣蝉）"},
  {"start": 47, "length": 2, "reason": "名词（长吟）"},
  {"start": 53, "length": 2, "reason": "名词（黄蜂）"}
]
```

> 注释：选「菜畦」「皂荚树」「桑椹」这种**鲁迅时代文学语料专有词**（低频），不选「的」「在」等虚词；段首「不必说」按规则 3 必选其一；密度 ≈ 25/89 ≈ 28%，偏低但合理（此段虚词密度高）。

### Example 2: 技术段落（专有名词密集）

**Input**：
```
Transformer 架构的核心是自注意力机制（Self-Attention），它通过 Query、Key、Value 三个矩阵计算序列内各位置的关联强度。
```

**Expected Output**：
```json
[
  {"start": 0, "length": 10, "reason": "专有名词（Transformer 架构）"},
  {"start": 15, "length": 1, "reason": "段首补足"},
  {"start": 17, "length": 5, "reason": "专有术语（自注意力机制）"},
  {"start": 24, "length": 14, "reason": "英文术语（Self-Attention）"},
  {"start": 43, "length": 5, "reason": "专有名词（Query/Key/Value）"},
  {"start": 53, "length": 1, "reason": "段中名词（矩阵）"},
  {"start": 63, "length": 1, "reason": "名词（关联）"}
]
```

> 注释：英文术语**整段高亮**（不要切分中英），因为眼睛对 Latin script 切换有 saccade 成本。

### Example 3: 短对话段落（高密度 + 段尾反转）

**Input**：
```
"我不去了。"她说。
```

**Expected Output**：
```json
[
  {"start": 1, "length": 3, "reason": "段首 + 实词（不去）"}
]
```

> 注释：极短段落，按规则 4 + 6：**段尾的引号和"她说"省略**；密度 3/8 = 37%。

## 4. 失败模式 & 防御 prompt 补充条款

| 失败模式 | 触发条件 | 防御补充条款 |
|---------|---------|------------|
| 偏移数错（特别是引号/全角字符） | 段落含 `"` `「」` `（` `""` 等 | 加："全角字符按 1 个 code unit 计数，标点不计入 start" |
| 高亮把整个长句涂满 | LLM 偷懒 | 加："禁止单次高亮 length > 6 字符" |
| 段间位置不重置 | 整篇 prompt | 加："每段 start=0 独立计数" |
| 输出多余文字（解释/道歉） | LLM 习惯 | 在 system message 加："只输出 JSON，不输出任何解释、致歉、markdown 包裹" |
| 英文/数字算错偏移 | 跨语言段落 | 加："英文单词整体计 1 个或多个 code unit，按 JS slice 行为" |
| 词性判断错误 | 罕见词 | 加："如果不确定是虚词还是实词，按实词处理（高亮）" |

---

## 5. 不放进 prompt 的"反规则"（防止 LLM 过度优化）

> 这些是**理论有争议**或**实验未验证**的，不写进主 prompt，避免污染 baseline。

- ❌ "在标点后高亮"（标点后就是下一句开头，**段首规则已覆盖**，重复会让 LLM 双重加权）
- ❌ "高亮数字和百分比"（无证据表明数字本身是认知瓶颈，**留给规则 1 专有名词兜底**）
- ❌ "避免高亮第一人称"（"我"虽然是高频虚词，但在叙事文中可能是关键 → **交给词频判断**）
- ❌ "高亮在名词前 1 字"（与 ORP 规则重复，**且容易让 LLM 偏移数错**）

## 6. 与原版 TillGlance `nlphl` 的差异声明

> 这点给 `agent-4-few-shot-iter` 参考。

| 维度 | 原版 `nlphl` | 我们的 prompt |
|------|-------------|--------------|
| 实现 | NLP 服务（黑盒） | LLM prompt |
| 输入 | `contentDict` 整篇 | 单段 / 整篇 |
| 输出 | `[start, len, start, len, ...]` 扁平数组 | JSON 数组 of objects |
| 高亮粒度 | 单字（每个字单独包 wrapper） | 词组 / 短语（更符合语义） |
| 强度调节 | `highlight_density` 滑块 | 通过 meta prompt 的"覆盖率"参数 |

**一个关键假设**：原版的"逐字高亮"在生产 UI 上**会让眼睛不知道看哪里**（每个高亮都是 35% 透明度的 span，**只有第一个字和最后一个字之间的对比有提示作用**）。我们改成"词组高亮"应该**不会比原版差，且可能更好**——这需要 `agent-4-few-shot-iter` 做对照实验验证。

## 7. A/B 实验设计（给 `agent-4-few-shot-iter` 留口子）

```
A 组：原版 nlphl 输出（baseline）
B 组：本 prompt v1 输出
C 组：本 prompt v1 + 删掉规则 6（中文反转 wrap-up）→ 看规则 6 的边际贡献
D 组：本 prompt v1 + few-shot 3 个例子
E 组：只给 few-shot 不给规则
```

**指标**：
- 覆盖率（高亮字符 / 段落总字符）→ 应在 30-50%
- 段首高亮命中率（首 25% 位置是否有高亮）→ 应 > 80%
- 偏移数准确率（与 ground truth 比对）→ 应 > 95%
- 与原版 nlphl 的重合率（IoU）→ 作为 sanity check，**不期望太高**（20-40% 合理，因为 prompt 自由度高）

---

## 8. 实施注意事项

1. **prompt 不要塞太多规则**：超过 10 条 LLM 注意力会分散，**10 条是上限**。本设计 10 条正好
2. **规则之间优先级要明确**：本设计用"1/2/3..."编号，且把高确定性的放前面
3. **few-shot 例子不超过 5 个**：token 成本和效果拐点
4. **每段独立 prompt 比整篇一次 prompt 更稳**：避免 LLM 在长 context 里"偷懒"（看到后面忘了前面）
5. **必须做 max_tokens 限制**：长文章的 MiniMax-M3 batching 已经在 web-mvp 里实现（见 `todo.md` 的 MiniMax Long Article Reliability 章节），沿用即可
